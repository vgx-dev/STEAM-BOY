# STEAM-BOY パフォーマンス・リファクタリング 要件定義書

- 文書バージョン: 1.0
- 作成日: 2026-06-24
- 対象: `STEAM-BOY.html`（単一HTMLファイル / 約433KB / 7,081行）
- 区分: 非機能（パフォーマンス）改善を主目的とした内部リファクタリング

---

## 1. 背景・目的

STEAM-BOY はPDF図面上に蒸気配管（Source / Valve / Trap / Drain / Split / Merge /
Reducer / HeatEx / Safety / Pgauge）を重ねて作図し、流体（STEAM / WATER / DRAIN）の
論理シミュレーション・異常検知・操作手順書生成・シーケンス再生までを行う
シングルファイルPWAである。

機能面はほぼ仕上がった一方、規模の大きい図面では描画・操作の重さが体感される。
本リファクタリングの目的は、**既存機能・UI・操作結果を一切変えずに、実行時パフォーマンス
（特にFPS）を引き上げる**こと。コード構造の整理はパフォーマンス改善の手段として
必要な範囲で行う。

### 重点シナリオ（優先度順）
本要件は以下の体感の重さを解消することを最優先とする。

1. **大規模図面の操作時**（ノード/エッジ多数でのドラッグ・ズーム・スクロール・弁操作）
2. **アニメーション再生中のFPS**（流体充填/排出アニメ、異常レーダー、シーケンス再生）
3. 上記に付随する全般的な応答性

## 2. 現状アーキテクチャと既存最適化（現状分析）

単一ファイル内に名前空間オブジェクトで責務分割されている。

| モジュール | 責務 |
|---|---|
| `Store` | 状態（nodes/edges/config/history）、ID→実体/隣接リストの遅延インデックス |
| `SimulatorCore` | 流体・圧力の純粋計算（副作用なし） |
| `SimWorker` | nodes+edges > 2000 のとき計算をWeb Workerへオフロード |
| `Simulator` | 計算＋`Anomaly.detect`＋`SealedDetector.compute`の取りまとめ |
| `Renderer` | SVGの差分レンダリング（`renderAll`）、マトリクス描画 |
| `Animator` | 充填/排出のWeb Animations、ゴーストパスの生成・破棄 |
| `Culling` | ビューポート外エッジのアニメ停止・`display:none`化 |
| `Minimap` | オフスクリーンcanvasキャッシュ＋レーダー波紋のrAF合成 |
| `Anomaly` / `SealedDetector` | 異常検知・密閉空間検知（無向BFS） |
| `OperationManual` / `Sequencer` / `Share` / `MasterTable` 等 | 手順書・再生・共有・一覧UI |

### 既に実装済みの最適化（重複実装を避けるため明記）
- **差分レンダリング**: エッジ/ノードごとに見た目入力の「シグネチャ」を算出し、不変なら
  DOM更新・パス再構築を丸ごとスキップ（`Renderer._edgeSig` / `_nodeSig`）。
- **属性レベルDirty Checking**: `setAttr` が現値と同一なら `setAttribute` を呼ばない。
- **隣接リスト化**: BFS・流入出集計の O(E) 全走査を O(deg) 参照へ置換。
- **計算オフロード**: 大規模時のみ Worker（閾値 `SimWorker.THRESHOLD = 2000`）。
- **ビューポートCulling**: 閾値 `DISPLAY_THRESHOLD = 300` で画面外を `display:none`。
- **perf-liteモード**: 閾値 `PERF_LITE_THRESHOLD = 500` で重い常時アニメ（ノードの
  drop-shadow breathing）等を停止、蒸気エッジのグラデを単色化。
- **rAFバッチ**: 高頻度イベントは `Renderer.scheduleRender` で1フレーム1回に間引き。
- **Minimap**: 静的グラフをオフスクリーンへキャッシュし、毎フレームは `drawImage`＋
  波紋＋ビューポート枠のみ合成。

### 残存ボトルネック仮説（本リファクタリングの主対象）
> ※確定は §7「計測」フェーズで行う。以下は着手前の仮説。

1. **`renderAll` がほぼ全操作の終端で呼ばれる O(N+E) フルパス**
   差分でDOM書込はスキップされるが、毎回 全ノード/全エッジのループとシグネチャ文字列
   生成・`Store.invalidateIndex()`・マトリクス再構築が走る。大規模図面では
   「変化が1要素でも全要素を舐める」コストが残る。
2. **連続アニメーションのFPS（SVG/CSSアニメの本数）**
   アクティブなエッジ数に比例してCSSフローアニメ（dash移動）が走る。SVGは要素数が
   増えるとPaint/Compositeコストが線形〜超線形に効く。大規模＋多数アクティブ時が最悪ケース。
3. **`Animator.apply` のDOMチャーン**
   排出アニメで非アクティブ化エッジ1本につきゴーストパス2本を `overlayLayer` に
   生成→`setTimeout`で破棄。一斉非アクティブ化時に大量の生成/GCが発生し得る。
4. **Culling/perf-lite/Worker の各閾値が固定**
   端末性能・図面規模に対して固定値（300/500/2000）。中規模かつ低速端末で
   「閾値未満だが重い」帯が存在し得る。
5. **大量の文字列連結シグネチャ**
   `renderAll` 内で要素ごとにシグネチャ文字列を生成（GC負荷）。
6. **シーケンス再生・弁プレビュー時の再計算頻度**
   `Simulator.run` → `Anomaly` → `SealedDetector` → `renderAll` の一括が
   操作のたびに同期で連鎖（小規模時）。

## 3. スコープ

### 3.1 対象（In Scope）
- `STEAM-BOY.html` 内の JS 実行時パフォーマンス改善（描画・計算・アニメ・イベント処理）。
- 上記に必要な範囲でのモジュール内部のコード整理・重複排除・関数分割。
- 閾値・ヒューリスティクスの動的化／チューニング。
- 計測の仕組み（任意起動のFPS/処理時間プローブ）の追加。

### 3.2 対象外（Out of Scope）
- **配布形態の変更**: 単一HTML維持。ビルド工程・モジュール分割（ファイル分割）・
  バンドラ導入は行わない（確定事項）。
- **機能・UI・操作結果・見た目の変更**（意図的変更は一切しない）。
- 新機能追加、シミュレーションロジックの計算結果の変更。
- `manifest.webmanifest` / `sw.js` の運用変更を伴う改修。
- 外部依存の入れ替え（pdf.js のバージョン変更等）。

## 4. 機能要件（＝機能維持の要件）

リファクタリング後も以下がリファクタリング前と**完全に同一**であること。

- FR-1: 作図操作（ノード/エッジ/中継点の追加・移動・削除・接続・矩形選択・複数ドラッグ）。
- FR-2: 流体・圧力シミュレーション結果（各エッジ/ノードの `_sim` 値、色、向き、残留、密閉）。
- FR-3: 異常検知（孤立・行き止まり・DANGER各種）とバッジ/パネル/ビネット表示。
- FR-4: アニメーション挙動（充填・排出のカスケード順序、タイミング、見た目）。
- FR-5: 弁操作・Reducer/Pressure セレクタ・操作マトリクスの内容と並び。
- FR-6: 操作手順書（OperationManual）・印刷・シーケンス再生・共有(Share)・一覧(MasterTable)。
- FR-7: 永続化（localStorage 保存/復元）、Undo/Redo、ミニマップ、ズーム/パン。
- FR-8: PWA としての起動・オフライン動作・オンボーディング・サンプル図面。

## 5. 非機能要件（パフォーマンス目標）

> 数値は計測ベースライン確定後に確定する暫定目標。代表端末・代表図面で測定する。

### 5.1 測定条件（ベンチ基準）
- 代表端末: （要確定。例: 中位ノートPC / 中位タブレット）
- 代表図面サイズ: 小（~100要素）／中（~600要素）／大（~2,000要素）／特大（>3,000要素）
- 測定対象操作: ドラッグ追従、ズーム、スクロール、弁ON/OFF（アニメ込み）、シーケンス再生、起動。

### 5.2 目標値（暫定）
- NFR-1 **アニメ再生中FPS**: 大規模図面のアニメ再生中に **平均55fps以上 / 最低45fps以上**
  （現状ベースライン比で改善。ベースライン値は計測後に追記）。
- NFR-2 **ドラッグ追従**: 大規模図面でのノードドラッグ時、入力から描画反映まで
  **1フレーム（16.7ms）以内** に収め、ロングフレーム（>50ms）を発生させない。
- NFR-3 **弁操作の応答**: 弁ON/OFFのクリックから再計算＋描画開始まで **100ms以内**
  （Worker利用時はメインスレッドをブロックしない）。
- NFR-4 **入力レイテンシ**: 高頻度入力（スライダー・ドラッグ）でメインスレッドの
  ロングタスク（>50ms）を発生させない。
- NFR-5 **メモリ**: アニメ多数回再生後もゴースト等のDOM/リスナーがリークしない
  （生成数と破棄数が一致）。
- NFR-6 **起動**: 機能・初期描画の完了が現状同等以下（退行させない）。

### 5.3 その他
- NFR-7: 機能退行ゼロ（§6 受け入れ基準で担保）。
- NFR-8: 単一ファイルサイズを著しく増やさない（リファクタで増加する場合は理由を明記）。

## 6. 受け入れ基準

- AC-1: §4 の全機能要件について、代表シナリオで**改修前後の挙動差分なし**を確認
  （手動チェックリスト＋可能なら保存データのスナップショット比較）。
- AC-2: §5 の各 NFR について、改修前ベースラインと改修後を同一条件で計測し、
  目標達成または明確な改善を示す（数値を本書 §9 に追記）。
- AC-3: シミュレーション計算結果（`_sim`）が代表図面で改修前とバイナリ等価
  （回帰用のゴールデンデータで比較）。
- AC-4: コンソールエラー・未破棄リスナー・未破棄DOMノードが増えていないこと。

## 7. 改善方針（候補施策・実装順は §8）

各施策は「効果が高く・リスクが低い」ものを優先。確定は計測結果による。

### A. 計測基盤（最初に着手）
- A-1: 軽量プローブ（FPSメータ、`renderAll`/`Simulator.run` 等の処理時間ログ）を
  デバッグフラグ（例: URLパラメータ / localStorage）で有効化できるようにする。
  本番表示・既存挙動には影響させない。
- A-2: 代表図面・代表操作のベースライン値を採取し本書に記録。

### B. 描画パス（最大の狙い）
- B-1: **ダーティ集合ベースの部分レンダリング**。「変化した要素ID集合」を Store 側で
  収集し、`renderAll` のフルループを変化要素のみのループへ縮小（全要素シグネチャ走査を回避）。
  ※ 既存のシグネチャ機構を活かしつつ「ループ対象自体」を削減するのが要点。
- B-2: マトリクス/ミニマップ/異常パネルの再構築を、関連状態が変化したときだけに限定。
- B-3: シグネチャの文字列連結を数値ハッシュ等の低GC方式へ（効果が出る場合のみ）。

### C. アニメーション／FPS
- C-1: 同時アニメ本数の上限制御（Culling と連携し、画面内かつ上限内のみアニメ。
  見た目の差が出ない範囲で）。
- C-2: `Animator` のゴーストパスを **プール再利用**（生成/GCの削減）。
- C-3: 連続アニメは可能な限り CSS/`transform`・compositor合成に寄せ、レイアウト/Paintを誘発する
  プロパティ更新を削減。
- C-4: Minimap レーダー等の rAF を、必要時のみ稼働・タブ非アクティブ時停止（既存方針の徹底）。

### D. 計算
- D-1: 弁プレビュー等の高頻度な `_baseRun` 呼び出しを必要最小限にデバウンス/メモ化。
- D-2: Worker 常駐化と差分入力転送の検討（毎回の全 nodes/edges シリアライズ削減）。
- D-3: `Anomaly`/`SealedDetector` を、グラフ構造が不変なら再利用できる部分はキャッシュ。

### E. 閾値・ヒューリスティクスの適応化
- E-1: Culling / perf-lite / Worker の各閾値を、実測FPSや端末性能に応じて適応調整
  （固定値の見直し。挙動の見た目は維持）。

### F. コード整理（B〜Eの副産物として必要な範囲）
- F-1: ホットパス（`renderAll`・`Animator`・`SimulatorCore`）の可読性確保と重複排除。
  ファイル分割はしない（単一HTML維持）。

## 8. 進め方（フェーズ）

1. **Phase 0 — 計測基盤とベースライン**（施策A）。回帰用ゴールデンデータ整備。
2. **Phase 1 — 描画パス改善**（施策B）。低リスク・高効果から。
3. **Phase 2 — アニメ/FPS改善**（施策C）。
4. **Phase 3 — 計算・Worker・閾値適応化**（施策D・E）。
5. 各フェーズ末で §6 受け入れ基準を再確認（機能退行ゼロを都度担保）。

各フェーズは独立した小さなコミットに分割し、いつでも切り戻せる状態を保つ。

## 9. リスクと対策

| リスク | 対策 |
|---|---|
| 描画最適化による表示差分（取りこぼし更新） | ダーティ集合の網羅性をテスト。差分レンダの後に「全再描画」デバッグコマンドで突合 |
| アニメ間引きによる見た目の劣化 | 画面内・上限内は従来同等を維持。閾値は計測で調整 |
| Worker差分転送の整合性崩れ | seq管理を踏襲し、不整合時はフル転送へフォールバック |
| 単一ファイルゆえの回帰範囲の広さ | フェーズ分割・小コミット・ゴールデン比較で局所化 |
| 体感はするが計測で出ない最適化に時間を使う | Phase 0 の計測を判断ゲートとし、効果の出る施策のみ実施 |
